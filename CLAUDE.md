# このリポジトリで作業するときの運用ルール

## 応答スタイル:深く考え、短く区切って答える

このリポジトリでは、1ターンが長時間(数時間〜24時間)になることを避ける。
考察は深くてよいが、**出力は細かく区切り、ユーザーに頻繁に報告する**こと。

### 必ず守ること

1. **最初に計画を共有する**
   - いきなり実装を始めない。
   - タスクを受け取ったら、まず 3〜7 個程度のサブタスクに分解し、箇条書きで提示する。
   - ユーザーの同意を得てから着手する。曖昧な点は `AskUserQuestion` で確認する。

2. **サブタスクごとに区切って手を止める**
   - 1 サブタスク完了 → 進捗と次にやることを 1〜3 行で報告 → ユーザーの応答待ち、を基本とする。
   - 「Phase 2 に進んで良いですか?」のように明示的に確認する。
   - 自走してよいのはユーザーが「まとめてやって」「最後まで進めて」と明示した場合のみ。

3. **長時間の独走を禁止する**
   - 1 ターン内のツール呼び出しが 20 回を超えそうな場合は、一度止めて状況を要約し、続行可否をユーザーに確認する。
   - ループ/リトライが発生したら即座に止め、原因と選択肢を提示する。
   - 無言でツールを連発しない。節目ごとに 1〜2 文のテキストを出す。

4. **深い思考は thinking で、出力は短く**
   - 推論・設計検討は thinking(拡張思考)側で行う。
   - ユーザーへの表示テキストは簡潔に。結論と次アクションだけを出す。
   - 進捗は TodoWrite で管理し、各 todo は完了と同時に `completed` にする(まとめて更新しない)。

5. **重い/不可逆な操作は必ず事前確認**
   - `git push --force`、`rm -rf`、DB マイグレーション、外部 API への書き込み等は、
     実行前に必ずユーザーに了承を取る。
   - 承認は指定された範囲に限られる。次回以降も同じ承認があるとみなさない。

### 禁止事項

- 何時間も黙って作業を続けること
- ユーザーの介入ポイントを作らずに大規模な変更をまとめてコミットすること
- リトライの無限ループ(3 回失敗したら止めて報告)
- 質問せずに要件を勝手に拡張すること(スコープクリープ)

### 目安

| 状況 | 取るべき行動 |
| --- | --- |
| タスクを受領 | 計画を提示し、同意を得る |
| サブタスク 1 件完了 | 1〜3 行で報告し、次の確認を求める |
| ツール呼び出しが 20 回を超えそう | 中間サマリを出して続行可否を確認 |
| 同じエラーで 3 回失敗 | 止めて原因と選択肢を提示 |
| 不可逆操作の前 | 必ず事前確認 |

ユーザーの時間を尊重すること。沈黙よりも「短い報告」を選ぶ。
